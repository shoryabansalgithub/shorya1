# Sprint 1 Architecture Freeze

**Status:** FROZEN  
**Phase:** 4 - Production Hardening  
**Scope:** Core Billing, Ledger, Event Driven Architecture, and Distributed Observability  

This document serves as the official, frozen architectural contract for Sprint 1. It outlines the rigid boundaries, data integrity rules, and distributed system mechanics that must not be bypassed or altered without a formalized RFC.

---

## 1. Final System Architecture
The application runs on a **NestJS API Backend** supported by **Prisma ORM**, bounded strictly to a **MySQL** datastore acting as the single source of truth. Background processing and task queues are strictly offloaded via **BullMQ** onto **Redis**. 

**Core Rules:**
- HTTP Requests are entirely stateless.
- Asynchronous jobs exclusively use BullMQ.
- Redis acts solely as a disposable cache and message broker; it never holds authoritative state.

---

## 2. Database Schema Guarantees
Database models are structurally coupled via explicit foreign keys to eliminate orphaned financial documents:
- **`LedgerTransaction`** MUST belong to an `Invoice` (`RESTRICT` delete).
- **`OutboxEvent`** creation is relationally coupled inside business transactions.
- **`InvoiceItem`** cannot exist without a valid parent `Invoice`.

Composite indexing (e.g., `[shopId, createdAt]`, `[idempotencyKey, shopId]`) natively prevents scan overhead on multi-tenant isolation lookups.

---

## 3. Ledger Immutability Rules
The core double-entry accounting layer enforces database-level immutability using MySQL Triggers:
- **`prevent_ledger_update`**: Once a `LedgerTransaction` record is inserted, it can **never** be updated. Any `UPDATE` command thrown at this table is natively rejected by the MySQL engine.
- **`prevent_ledger_delete`**: Financial history is permanent. Any `DELETE` command targeting a `LedgerTransaction` is natively blocked by the MySQL engine.

To correct a financial error, a new compensatory `LedgerTransaction` must be appended. Historical mutation is physically impossible.

---

## 4. Transactional Outbox Flow
To guarantee dual-write safety between MySQL and the message broker (BullMQ):
1. **Creation:** Inside the exact same `Prisma.$transaction` that deducts stock and creates an Invoice, an `OutboxEvent` record is inserted into MySQL with `status = PENDING`.
2. **Relay:** A CRON `OutboxRelayService` polls MySQL using `SELECT ... FOR UPDATE SKIP LOCKED`.
3. **Dispatch:** The Relay Worker translates the MySQL payload to a BullMQ `job` and executes `.addBulk()`.
4. **Resolution:** Upon successful enqueue to Redis, the MySQL event updates to `status = DONE`.

*Guarantees zero dropped events, even if Redis restarts violently during a transaction.*

---

## 5. BullMQ Architecture
Background orchestration requires extreme resiliency:
- **Consumer Idempotency:** Consumers check `AuditLog` via `eventId` to verify if the event was previously handled.
- **Deduplication:** The `OutboxEvent.id` maps directly to the BullMQ `jobId`, preventing Redis network retries from duplicating enqueues.
- **Retry Semantics:** 
  - *Transient Failures* (e.g., deadlocks) trigger exponential backoff.
  - *Permanent Failures* (e.g., syntax errors) throw `UnrecoverableError`, skipping retries and dropping straight into a Dead Letter Queue.

---

## 6. Redis as Disposable Cache Strategy
Redis handles **optimistic stock locking** and **queue transit**. It is deliberately designed to be entirely disposable.
- If Redis data corrupts, goes offline, or is completely flushed, the system falls back onto MySQL.
- Cache misses query MySQL natively.
- **Invariant:** If Redis implies there is stock but MySQL says otherwise, the Prisma transaction fails gracefully and enforces the MySQL constraint.

---

## 7. Idempotency Design
Every destructive or financial flow adheres to a multi-tiered idempotency structure:
- **Frontend Layer (`useIdempotencyKey`):** Upon component mount (not button click), a UUID is generated and mapped to `sessionStorage`. It survives browser refreshes and 504 network timeouts.
- **Backend Layer:** The `idempotencyKey` binds exclusively to the `Invoice`. A `PrismaClientKnownRequestError` (`P2002` duplicate collision constraint) intercepts any parallel duplicate requests arriving at identical milliseconds, throwing away the second thread and returning the exact invoice generated by the first.

---

## 8. Correlation ID Tracing Flow
Logs across HTTP requests, Cron jobs, and Worker events are seamlessly woven via Node `AsyncLocalStorage`.
- **Inbound:** `CorrelationIdMiddleware` sniffs `x-correlation-id` or generates a fresh `uuidv4()`.
- **Propagation:** Overrides on `ConsoleLogger` pull from `CorrelationContextService` implicitly. Every log emits the `correlationId` without business logic modification.
- **BullMQ:** The ID binds into the JSON `OutboxEvent` payload, relays through BullMQ data structures, and wraps the Worker execution context.
- **Security:** Logging engine recurses through objects and permanently converts values with `password`, `token`, `cookie`, `secret`, or `payment` keys to `[REDACTED]`.

---

## 9. Reconciliation System
A fail-safe `InventoryReconService` daemon operates independently of the billing application:
- Protected under `CronLockService` (Redlock), ensuring only one cluster pod operates at a time.
- Compares Redis cache states against native Prisma values.
- Executes silent `SET` repairs when anomalies are discovered.
- **Auditing (`ENG-502`):** All drifts dynamically spawn `InventoryDriftLog` models inside MySQL, permanently logging `redisValue`, `databaseValue`, and tracking state transitions (`DETECTED` -> `REPAIRED` / `FAILED`).

---

## 10. Failure Recovery Matrix

| System Failure Scenario | Automated Recovery Response |
| :--- | :--- |
| **Node / Pod Crash mid-transaction** | Prisma rolls back the transaction. No financial logic occurs. No orphaned data. |
| **Client loses internet after submit** | The browser idempotency key ensures the next retry simply loads the already-completed invoice. |
| **Redis node restart (Cache Wipe)** | Application gracefully reads directly from MySQL. The CRON Reconciliation syncs inventory cache dynamically. |
| **BullMQ Worker Crash** | The message remains locked in the queue or triggers exponential backoff. Processed upon restart. |
| **Parallel multi-pod Cron Execution** | `CronLockService` ensures only the node holding the Redis lock executes the logic. Others silently bypass. |
| **Malicious Request Headers** | `CorrelationIdMiddleware` regex drops excessively long or malicious IDs, injecting a fresh safe UUID instead. |
